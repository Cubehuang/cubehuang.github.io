3.2 rt_mutex 锁用法和持锁排查

互斥锁mutex只允许一个进程进入临界区，是和保护比较长的临界区，因为竞争互斥锁时进程可能休眠和再次唤醒，代价很高。

rt_mutex实时互斥锁：
实时互斥锁是对互斥锁的改进，实现了优先级继承（priority inheritance）解决了优先级反转（priority inversion）问题。

  • 什么是优先级反转问题？

    假设进程1的优先级低，进程2的优先级高。进程1持有或持锁，进程2申请互斥锁，因为进程1邮件占有互斥锁，所以进程2必须休眠等待，导致优先级高的进程2等待优先级低的进程1
    如果存在进程3，优先级在进程1和进程2之间，情况更糟糕，假设进程1仍然持有互斥锁，进程2正在等待。进程3开始运行，因为它的优先级比进程1高，所以它可以抢占进程1，导致进程1持有互斥锁的时间延长，进程2等待的时间更长。
    优先级继承则可以解决优先级反转问题。如果低优先级的进程持有互斥锁，高优先级的进程申请互斥锁，那么把持有互斥锁的进程优先级临时提升导申请互斥锁的进程的优先级。在上面的例子中，把进程1的优先级提升到了进程2的优先级，防止进程3抢占进程1，使得进程1尽快执行完成临界区，减少进程2的等待时间。

实时互斥锁的定义如下：

可以通过crash 工具中struct rt_mutex.owner 锁地址 获取到持锁进程
申请实时互斥锁的函数如下：

  • void rt_mutex_lock(struct rt_mutex *lock)
    申请实时互斥锁，如果锁被占有，进程进入不可中断休眠
  • int rt_mutex_lock_interruptible(struct rt_mutex *lock)
    申请实时互斥锁，如果锁被占有，进程进入可中断休眠
  • int rt_mutex_timed_lock(struct rt_mutex *lock, struct hrtimer_sleeper *timeout)
    申请实时互斥锁，如果锁被占有，进程睡眠等待一段时间。
  • int rt_mutex_trylock(struct rt_mutex *lock)
    申请实时互斥锁，如果申请成功，返回1，如果锁被其他进程占有，进程不等待，返回0.

3.3 rcu 锁用法和持锁排查

from:《Linux 内核深度解析》-余华兵

  • RCU(Read-Copy Update)的意思是读-复制更新，它是根据原理命名的。写者修改对象的过程是：首先复制生成一个副本，然后更新这个副本，最后使用新的对象替换旧的对象。在写者执行复制更新的时候读者可以读数据。
  • 写者删除对象，必须等到所有访问被删除对象的读者访问结束，才能执行销毁操作，RCU的关键技术是怎么判断所有读者已经完成访问。等待所有读者访问结束的时间称为 宽限期（grace period）
  • RCU的优点是读者没有任何同步开销：不需要获取任何锁，不需要获取任何锁，不需要执行院子指令不需要执行内存屏障。但是写者的同步开销比较大，写者需要延迟对象的释放，复制被修改的对象吗，写者之间必须使用互斥锁。

一共有多个版本的RCU：

  • 不可抢占RCU（RCU-sched）不允许进程在读端临界区被其他进程抢占。最初的RCU就是不可抢占RCU
  • 加速不可抢占RCU（RCU-bh），在内涵版本2.6.9版本中引入，针对不可抢占RCU的改进，在软中断很多的情况下可以缩短宽限期。
  • 可抢占RCU（RCU- preempt），也称为实时RCU，在内涵2.6.26版本中引入，可抢占RCU允许进程在读端临界区被其他进程抢占（我们的此问题就是此种RCU）

技术原理：
RCU术语：

  • 读端临界区（Read-Side Critical Section）：读者访问RCU保护的对象的代码区域。
  • 静止状态（quiescent state）：处理器的执行状态吗处理器没有读RCU保护的对象，也可以理解为读者禁止，没有访问临界区。
  • 候选静止状态（candidate Quiescent state）：读端临界区以外的所有的点都是候选静止状态。
  • 可被观察的静止状态（Observed Quiescent state）：一部分候选静止状态可以很容易观察到，因为它们和内核的某个事件关联，通过内核的事件可以观察到。把这些容易观察到静止状态称为可被观察的静止状态。内核的RCU使用的静止状态是可被观察的静止状态，不可抢占的RCU通过时间“进程调度调度进程”观察到静止状态，加速版不可抢占RCU通过“执行完软中断”观察到静止状态。
  • 宽限期（grace preiod）：等待所有处理器上的读者退出临界区的时间被称为宽限期。如果所有处理器都至少经历一个静止状态，那么当前宽限期结束，新的宽限期开始。当前宽限期结束的时候，写者在当前宽限期及仪器注册的销毁对象的回调函数就可以安全的执行。

宽限期又分为：正常宽限期（normal grace period）和加速宽限期（expedited grace period）调用函数synchronize_rcu_expedited()等待宽限期结束的时候，会向其它的处理器发送处理器中断（Inter-Processor_Interrupt，IPI）请求，强制产生静止状态，使宽限期快速结束，我们把强制开始结束的宽限期称为加速宽限期。

  • 可抢占RCU
    可抢占RCU（RCU-preempt）允许进程在读端临界区里面被其他进程抢占，从 函数rcu_read_lock() 标记读端临界区开始，使用rcu_read_unlock()标记读端临界区结束。
    可抢占的RCU观察静止状态，需要考虑一下因素：
  • 进程在读端临界区里面被其他进程抢占以后，可能被迁移到其他处理器：可能是同一个分组的其他处理器，也可能是其他分组的处理器。造成的后果就是：进程可能在一个处理器上进入读端临界区，在同一个分组的另一个处理器上退出读端临界区；进程可能在一个分组中进入读端临界区，在另一个分组中退出读端临界区。
  • 读端临界区可能嵌套

可抢占RCU观察静止状态的实现方案如下：

  1. 不是以处理器为单位观察静止状态，而是以最底层分组为单位观察静止状态。不可抢占RCU和加速版不可抢占RCU都是以处理器为单位观察静止状态。
  2. 进程需要记录读端临界区的嵌套层数，进入读端临界区时候把嵌套层数加1，退出读端临界区的时候吧嵌套层数减1。当嵌套层数变成0的时候，说明进程已退出最外层的读端临界区。
  3. 当进程在读端临界区里面被其他进程抢占的时候，把进程添加到分组的阻塞进程链表中，并且记录分组。
  4. 当进程退出最外层的读端临界区的时候，把进程从分组的阻塞进程链表中删除。如果分组的阻塞进程链表变成为空，那么观察到一个静止状态，注意这里的分组是第三部中记录的分组，不是进程当前所在的处理器的分组。换句话说，在分组n中进入读端临界区的进程，不管是不是在分组n中退出读端临界区，都看作在分组n中退出读端临界区。当在分组n中进入读端临界区的所有 进程退出读端临界区的时候，分组n观察到一个静止状态。
  • rcu_read_unlock_special:记录读端临界区的嵌套次数

3.5 主线程和子线程退出流程-探究

结论：当使用kill -9 主线程，会将信号值填入线程task.signal.shared_pending.signal 中（所有线程共享该结构），由于SIGKILL特殊，所有线程组中的线程的task.pending.signal中都会添加SIGKILL信号(zap_other_threads)，唤醒所有线程组线程处理该SIGKILL信号。“唤醒”：实则设置task.thread_info.flags设置为 TIF_SIGPENDING，后才唤醒线程
随后线程被唤醒，会进行获取信号进行信号检查，然后处理信号，由于Android进程中SIGKILL并没有注册用户态的信号处理函数，并不用返回用户态处理信号处理函数(get_signal)，非主线程的其他线程组线程会直接走 do_exit 退出，其后的exit_notify会将线程的tsk->exit_state = EXIT_DEAD，然后在do_task_dead中设置 tsk->__state 为TASK_DEAD ，线程就退出了。
主线程特殊性：刚刚的流程中，对于主线程，在exit_notify主线程先tsk->exit_state = EXIT_ZOMBIE，然后在do_task_dead中设置 tsk->__state 为 TASK_DEAD 。此时主线程为所谓的 Z（僵尸）状态，主线程还未完全退出，主线程的tsk->exit_state 设置EXIT_DEAD，这个线程才会真正退出。
那么主线程什么时候设置 exit_state = EXIT_DEAD呢？

  • 在do_group_exit流程中 主线程先设置 tsk->exit_state = EXIT_ZOMBIE 后，如果此时线程组中子线程都退出了thread_group为空，那么此时（autoreap = thread_group_empty(tsk) && do_notify_parent(tsk, tsk->exit_signal)）会唤醒父进程，并向父进程发送 SIGCHLD ，让父进程调用waitpid接口，在内核中等待唤醒，父进程被唤醒后，(wait_consider_task)会遍历检查所有子进程（task.children）,(wait_task_zombie)对于没有子线程的主线程（thread_group为空）则会将tsk->exit_state 由 EXIT_ZOMBIE 改为EXIT_DEAD，并且释放该主线程，至此主线程才退出
  • 在release_task流程中，如果release的是线程组中最后一个子线程并且主线程为Z状态，那么也会do_notify_parent 通知父进程
    特别注意：如果有子线程未退出，那么主线程将会一直 Z 状态。正如问题中遇到的 子线程D状态或者子线程内核态陷入死循环，关闭中断等，无法返回用户态无法处理信号，无法退出，就会导致主线程长时间Z状态.进而影响到整个进程退出。
